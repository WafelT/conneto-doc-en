Code Writing Patterns

This page is intended to showcase selected and frequently used code solutions in the project for the purpose of standardization and maintaining a unified coding style.

As a result, increased predictability and reduced time for understanding the developed functionality are expected.

Toast Message Declaration Template

The Toast message is declared on the page of exploitation, via MyToast component, it get's a unique name parameter, the calling of message is executed from eventBus using his $emit

An exception is the Toast message used to display unsuccessful responses from the server that are not included in the scenarios for using the functionality, in this case we can directly access the $toast object

Explanations: for example, Toast message of permissions absence are declared through the component since this is a part of expected script of usage, errors from try/catch are triggered in case of invalid response, so they are called using the $toast object

Icon Declaration Template

For precise styling of the icon based on its state, the class .i-background is used, which has modification classes --fill, --stroke. These modification classes are used to orient in CSS styling and are applied only to dynamic icon tags. They can also be used together.

Endpoint Declaration Template

End-points are declared through the $axios key from the nuxt context. The first argument needs to be specified as an object with all the necessary parameters.

End-points are declared only in action functions in store files.

Props Declaration Template

To declare props in the project, the VueTypes plugin is used. It allows specifying the necessary data for the props in a compact single line.

Advantage: Compactness

Getters Declaration Template

Data from the store should be accessed in the component using the following format, utilizing the spread operator for the auxiliary function mapGetters.

Advantage: This notation allows easily understanding from which store file the data is being accessed.

Action Declaration Template

Action functions should be called directly from the component context.

Advantage: This allows easily distinguishing an action function from a regular component method, speeding up debugging.

Mutations Declaration Template

Mutation functions are declared in uppercase and separated by an underscore. They are used exclusively within store files!

CSS Styles Declaration Template

CSS styles are declared in .scss files, and component-specific styles are declared only in cases of extreme necessity.

This approach maintains order in the project and eliminates conflicts in style priorities.
